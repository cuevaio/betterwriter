# Streaming Architecture Refactor

## Problem
The API hangs indefinitely even when data exists in DB. Redis entity locks get stale ("running" but generation is done/crashed), causing POST to return "already-running" without checking DB. iOS then tails a dead SSE stream forever.

## Goals
1. Server always returns data from DB when it exists (DB check BEFORE Redis)
2. JSON response (200) for existing data, SSE stream (202) for generation
3. Single POST request (eliminate separate GET endpoint)
4. iOS pre-fetches both reading + prompt after auth
5. Typewriter animation for both JSON (fast) and SSE (real-time) data

## Implementation Order

### Step 1: Server — durable-stream.ts simplification
**File:** `apps/web/lib/ai/durable-stream.ts`

- Change `DEFAULT_TTL_SECONDS` from 86400 (24h) to 300 (5 min)
- Change `STALE_RUNNING_MS` from 10 min to 2 min
- In generation runners: release entity lock 5s after emitting `complete` event (no longer need lock to persist for GET since GET is being removed)

### Step 2: Server — new inline SSE response with DB fallback
**File:** `apps/web/lib/api/durable-sse.ts`

Add `createInlineSSEResponse(request, streamId, userId, dayIndex, kind)`:
- Tails Redis events same as current `createDurableSSEResponse`
- **NEW: DB fallback** — every 3s of idle, queries DB for completed data
- 30s hard timeout — if no events and no DB data, emit error event
- 5s wait for stream metadata to appear (QStash has latency)
- Returns `202` with SSE headers

### Step 3: Server — rewrite POST handlers (readings + prompts)
**Files:**
- `apps/web/app/api/readings/generate/stream/route.ts`
- `apps/web/app/api/prompts/generate/stream/route.ts`

New POST flow:
```
1. QStash signature? → delegate to workflowPOST (unchanged)
2. requireUserId
3. DB CHECK FIRST: query entries for readingBody/writingPrompt
   → exists? Return 200 JSON { entry: {...} }
4. Check Redis lock (resolveActiveStream)
   → running? Return 202 SSE (tail with DB fallback)
   → completed? Re-check DB, release stale lock if empty
5. Acquire lock, dispatch via QStash (prod) or in-process (dev)
   → Return 202 SSE (tail own stream with DB fallback)
```

**DELETE the GET handler entirely** — no longer needed.

Wrap QStash publish in try/catch: if it fails, release lock and return 500.

Add `serializeEntry(row)` helper for consistent JSON response shape.

### Step 4: Server — update generation runners for lock release
**Files:**
- `apps/web/app/api/readings/generate/stream/route.ts` (runReadingGeneration)
- `apps/web/app/api/prompts/generate/stream/route.ts` (runPromptGeneration)

After emitting `complete` event, schedule lock release after 5s grace:
```typescript
setTimeout(async () => {
  await releaseEntityLock(userId, kind, streamId);
}, 5000);
```

### Step 5: iOS — new APIClient methods
**File:** `apps/ios/betterwriter/betterwriter/Services/APIClient.swift`

Add `GenerateResult<T>` enum:
```swift
enum GenerateResult<T: Sendable>: Sendable {
  case immediate(T)       // JSON from DB (200)
  case stream(AsyncThrowingStream<T, Error>)  // SSE generation (202)
}
```

Add `generateReading() -> GenerateResult<ReadingStreamEvent>`:
- POST to `/api/readings/generate/stream`
- Check status code: 200 → accumulate bytes, decode JSON, return `.immediate`
- 202 → parse SSE bytes inline, return `.stream`

Add `generatePrompt() -> GenerateResult<PromptStreamEvent>`:
- Same pattern for prompts

Add `parseSSEBytes(bytes:label:parseFrame:)`:
- Extracted SSE line parser from current `streamSSE`
- Works on `URLSession.AsyncBytes` directly
- No retry logic (caller handles retries)

**Delete**: `startReadingStream()`, `streamReading()`, `startPromptStream()`, `streamPrompt()`, `streamSSE()`, `StartStreamResponse`

### Step 6: iOS — PrefetchStore
**File:** `apps/ios/betterwriter/betterwriter/Services/PrefetchStore.swift` (NEW)

`@MainActor @Observable` singleton with:
- `reading: FetchState<EntryResponse>` (.idle | .loading | .ready | .streaming | .failed)
- `prompt: FetchState<String>` (same)
- `prefetch()` — fires both `generateReading()` and `generatePrompt()` in parallel
- `reset()` — cancels tasks, resets to `.idle`
- Guard against duplicate calls (no-op if already loading/ready)

### Step 7: iOS — RootView pre-fetch trigger
**File:** `apps/ios/betterwriter/betterwriter/Views/RootView.swift`

In `.task`, after `ensureProfile()` and before `advanceState()`:
```swift
PrefetchStore.shared.prefetch()
```

In `onChange(of: scenePhase)` when `.active`:
- Reset and re-prefetch if day has advanced

### Step 8: iOS — ReadView simplification
**File:** `apps/ios/betterwriter/betterwriter/Views/ReadView.swift`

Rewrite `loadReading()`:
1. Check SwiftData cache → if `readingBody` exists, return
2. Check `PrefetchStore.shared.reading`:
   - `.ready(entry)` → save to SwiftData, typewriter animate (fast mode)
   - `.streaming(stream)` → consume events, typewriter animate (normal mode)
   - `.failed/.idle/.loading` → call `generateReading()` directly (fallback)
3. One retry on failure (reset prefetch state, retry)

### Step 9: iOS — WriteView simplification
**File:** `apps/ios/betterwriter/betterwriter/Views/WriteView.swift`

Rewrite `loadPromptAndDraft()`:
1. Check SwiftData cache → if `writingPrompt` exists, return
2. Check `PrefetchStore.shared.prompt` (same pattern as ReadView)
3. Save prompt to SwiftData entry
4. One retry on failure

### Step 10: iOS — TypewriterAnimator fast mode
**File:** `apps/ios/betterwriter/betterwriter/Utilities/TypewriterAnimator.swift`

Add `batchSize` parameter to `wordChunks(from:batchSize:)`:
- `batchSize=1` for SSE deltas (current behavior)
- `batchSize=5` for JSON cached data (5 words at a time, 10ms delay → ~1.6s for 800 words)

Update `enqueueTypewriter(_:fast:)` in ReadView/WriteView:
- `fast: true` for pre-fetched/JSON data
- `fast: false` for SSE stream deltas
