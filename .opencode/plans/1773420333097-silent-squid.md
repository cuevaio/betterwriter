# Fix: Bonus reading shows old content instead of streaming new

## Root Cause

**Race condition** between fire-and-forget `PUT /api/entries` and the subsequent `POST /api/readings/generate/stream`.

Flow that triggers the bug:
1. User taps "DONE READING" in `BonusReadView`
2. `completeReading()` sets `readingCompleted = true` locally, fires PUT to server as **fire-and-forget**, and immediately calls `onComplete()` → transitions to DoneView
3. User taps "Read something" → new `BonusReadView`, POST to server
4. Server finds the old completed stream's entity lock (24h TTL, intentionally retained). Checks DB: `readingCompleted` is still `false` because the PUT hasn't arrived yet
5. Server returns `{ mode: "completed", entry: {OLD reading} }` instead of generating a new one

## Fix (2 files)

### 1. `apps/ios/betterwriter/betterwriter/Views/BonusReadView.swift` (primary)

**a) Add `isCompleting` state** (after line 20):
```swift
@State private var isCompleting = false
```

**b) Rewrite `completeReading()` (lines 287-311)** — move `onComplete()` inside the Task, after awaiting the PUT:
```swift
@MainActor
private func completeReading() {
  guard let entry = managedEntry, !isCompleting else { return }
  isCompleting = true

  entry.readingCompleted = true
  entry.needsSync = true
  do { try modelContext.save() } catch {
    print("BonusReadView: save completion failed: \(error)")
  }

  Task {
    do {
      _ = try await APIClient.shared.updateEntry(
        fields: ["readingCompleted": true, "isBonusReading": true]
      )
    } catch {
      print("BonusReadView: Failed to sync reading completion: \(error)")
    }
    // Navigate only after server acknowledges (or fails)
    onComplete()
  }
}
```

**c) Update `doneButton` (lines 137-145)** — show loading indicator, disable during completion:
```swift
private var doneButton: some View {
  Button(action: completeReading) {
    if isCompleting {
      ProgressView()
        .tint(WQColor.primary)
    } else {
      Text("DONE READING")
    }
  }
  .buttonStyle(WQOutlinedButtonStyle())
  .disabled(isCompleting)
  .padding(.horizontal, Spacing.contentHorizontal)
  .padding(.bottom, Spacing.l)
  .background(.ultraThinMaterial)
}
```

### 2. `apps/web/app/api/entries/route.ts` (complementary server-side)

**a) Add imports** (after line 13):
```typescript
import { getActiveStreamId, releaseEntityLock } from "@/lib/ai/durable-stream";
```

**b) Release entity lock after upsert** (between lines 166 and 168):
```typescript
if (allowedFields.readingCompleted === true) {
  const activeStreamId = await getActiveStreamId(userId, "reading");
  if (activeStreamId) {
    await releaseEntityLock(userId, "reading", activeStreamId);
  }
}
```

This ensures the next POST to generate a reading won't find a stale lock pointing to the completed reading.

## Why both fixes

| Scenario | iOS fix alone | Server fix alone | Both |
|---|---|---|---|
| Normal flow | Works (PUT before POST) | Works (lock released) | Redundant safety |
| PUT slow (>2s) | Still works (awaited) | Still works | Covered |
| PUT fails | User proceeds, SyncService retries | N/A | iOS proceeds gracefully |
| Multi-device | Not covered | Covered | Covered |
