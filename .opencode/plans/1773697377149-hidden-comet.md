# Fix: "DONE WRITING" button does nothing on Day 2+

## Root Cause

A race condition between `modelContext.save()` and SwiftData `@Query` propagation:

1. `WriteView.completeWriting()` sets `writingCompleted = true`, calls `modelContext.save()`, then immediately calls `onComplete()` ŌåÆ `advanceState()` in the **same RunLoop tick**.
2. `advanceState()` reads from `RootView`'s `@Query entries`, which hasn't re-fetched yet ŌĆö so `DayEngine.resolveCurrentPhase()` sees stale data (`writingCompleted = false`).
3. With stale data, `resolveCurrentPhase()` returns `.write(dayIndex: N, aboutDayIndex: N-2)` ŌĆö the **same phase** as the current one ŌåÆ `advanceState()` sees `resolved == currentPhase` ŌåÆ **no transition**.
4. The `.onChange(of: entries)` fallback (added in a previous fix) **never fires** because `@Model` classes use identity-based `Equatable` (`PersistentIdentifier` comparison). Mutating `writingCompleted` on an existing entry doesn't change array equality.
5. The only recovery is backgrounding and re-foregrounding the app (which triggers `scenePhase == .active` ŌåÆ `advanceState()` with eventually-fresh query data).

This affects all days, but Day 0/1 may appear to work due to shorter sessions or faster propagation.

## Fix

**Single change in `RootView.advanceState()`**: replace `Array(entries)` (stale `@Query`) with a fresh `modelContext.fetch()` call, which reads the current in-memory state including just-saved mutations.

### File: `apps/ios/betterwriter/betterwriter/Views/RootView.swift`

**Current** (line 210-225):
```swift
private func advanceState(profileOverride: UserProfile? = nil) {
    if case .freeWrite = currentPhase { return }
    if case .bonusRead = currentPhase { return }

    let effectiveProfile =
      profileOverride ?? resolvedProfile ?? profile

    let resolved = DayEngine.resolveCurrentPhase(
        profile: effectiveProfile,
        entries: Array(entries)    // ŌåÉ stale @Query data
    )
    if resolved != currentPhase {
        currentPhase = resolved
    }
}
```

**Fixed**:
```swift
private func advanceState(profileOverride: UserProfile? = nil) {
    if case .freeWrite = currentPhase { return }
    if case .bonusRead = currentPhase { return }

    let effectiveProfile =
      profileOverride ?? resolvedProfile ?? profile

    // Fetch entries directly from the model context to pick up
    // mutations that @Query hasn't propagated yet (same RunLoop tick).
    let freshEntries: [DayEntry]
    do {
        let descriptor = FetchDescriptor<DayEntry>(
            sortBy: [SortDescriptor(\DayEntry.dayIndex)]
        )
        freshEntries = try modelContext.fetch(descriptor)
    } catch {
        freshEntries = Array(entries)
    }

    let resolved = DayEngine.resolveCurrentPhase(
        profile: effectiveProfile,
        entries: freshEntries
    )
    if resolved != currentPhase {
        currentPhase = resolved
    }
}
```

Keep the existing `.onChange(of: entries) { advanceState() }` as a safety net for background mutations (sync service, etc.). It's harmless since `advanceState()` guards with `resolved != currentPhase`.

## Why this works

- `ModelContext.fetch()` reads from the unit-of-work cache, which includes all in-memory mutations (even unsaved ones). After `modelContext.save()` in `WriteView.completeWriting()`, a `fetch()` on the same `modelContext` will see `writingCompleted = true` immediately.
- The fix is deterministic ŌĆö no timing, no retries, no duplication of phase logic.
- `DayEngine.resolveCurrentPhase()` remains the single source of truth.
