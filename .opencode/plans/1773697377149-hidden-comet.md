## Bug 3 (THIS PLAN): Phase transition doesn't fire in RELEASE builds

### Symptoms
- **Simulator (DEBUG)**: tap "DONE WRITING" → screen transitions to Done/stats screen. Works.
- **Real device (RELEASE)**: tap "DONE WRITING" → haptic fires, nothing else happens. Screen stays on WriteView.

### Root Cause

The phase transition in `RootView.advanceState()` sets `currentPhase = resolved` (a `@State` property). The `ZStack` in the body uses a `switch` on `currentPhase` to render the appropriate view, with `.animation(.spring(...), value: currentPhase)` to animate transitions.

**In DEBUG builds**, the Inject framework's `.enableInjection()` wraps every view in `AnyView`, forcing SwiftUI to use **dynamic identity-based diffing**. This means SwiftUI always detects when one view type replaces another and plays the transition.

**In RELEASE builds**, `.enableInjection()` is an identity no-op (`return self`). SwiftUI uses **structural identity** from the `switch` cases. The `.animation(_, value:)` modifier is supposed to detect the `currentPhase` change and animate, but this pattern is known to be unreliable in RELEASE builds — especially when the state change comes from a child view's synchronous callback (`onComplete`). The implicit animation sometimes fails to trigger the view swap/transition.

**The fix**: Wrap the `currentPhase` assignment in an explicit `withAnimation` block inside `advanceState()`. This is the standard SwiftUI pattern to ensure transitions fire reliably regardless of build configuration.

### Files to modify

| File | Change |
|---|---|
| `apps/ios/.../Views/RootView.swift` | Wrap `currentPhase = resolved` in `withAnimation` |

### Implementation

In `RootView.advanceState()` (line ~234), change:
```swift
if resolved != currentPhase {
    currentPhase = resolved
}
```
to:
```swift
if resolved != currentPhase {
    withAnimation(.spring(response: 0.5, dampingFraction: 0.85)) {
        currentPhase = resolved
    }
}
```

The existing `.animation(.spring(...), value: currentPhase)` modifier on the ZStack can be kept as a fallback or removed (since `withAnimation` now drives the animation explicitly). Keeping it is harmless.
