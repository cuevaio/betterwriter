# Fix: "DONE WRITING" button does nothing on Day 1+

## Status: Phase 1 fix already committed (a049e84). Phase 2 below.

## Bug 1 (Phase 1 — DONE): Local UI transition fails

`advanceState()` read stale `@Query` data. Fixed by using `modelContext.fetch()` instead.

## Bug 2 (Phase 2 — THIS PLAN): Server never receives `writingCompleted = true`

### Root Cause

The `PUT /api/entries` call from `WriteView.completeWriting()` and `ReadView.completeReading()` does NOT send `dayIndex`. The server computes it via `getCurrentDayIndex(userId)`, which returns `max(completed dayIndex) + 1`.

This is fragile because:
- If ANY prior day's completion failed to sync, the server computes the wrong dayIndex, and the update lands on the wrong entry.
- Even in the happy path, the two sequential calls in `completeWriting()` create a problem: after `PUT /api/entries` sets `writingCompleted=true` on day N, `POST /api/user-input` computes `getCurrentDayIndex()` = N+1 and writes `writingText`/`writingWordCount` to the **wrong day** (N+1 instead of N).

Evidence from DB: Day 1 has `writingCompleted=0` despite the user tapping "DONE WRITING". The update either went to the wrong dayIndex or never reached the server.

### Fix: Send `dayIndex` from the client in `PUT /api/entries`

**Server changes:**

1. `apps/web/lib/api/schemas.ts` — Add optional `dayIndex` to `entryUpdateSchema`:
   ```typescript
   export const entryUpdateSchema = z.object({
     dayIndex: z.number().int().min(0).optional(),  // NEW
     // ... rest unchanged
   });
   ```

2. `apps/web/app/api/entries/route.ts` — Use client-provided `dayIndex` when present:
   ```typescript
   // Line 160: change from always computing to preferring client value
   const dayIndex = updates.dayIndex ?? await resolveTargetDayIndex(userId, updates);
   ```
   Also strip `dayIndex` from `allowedFields` so it isn't passed to `pickDefined` (it's used for targeting, not as an updatable field).

**iOS changes:**

3. `apps/ios/.../Views/WriteView.swift` — Add `dayIndex` to the updateEntry fields:
   ```swift
   _ = try await APIClient.shared.updateEntry(
     fields: [
       "dayIndex": dayIndex,           // ADD
       "writingText": userText,
       "writingWordCount": wordCount,
       "writingCompleted": true,
     ]
   )
   ```

4. `apps/ios/.../Views/ReadView.swift` — Add `dayIndex` to the updateEntry fields:
   ```swift
   _ = try await APIClient.shared.updateEntry(
     fields: [
       "dayIndex": dayIndex,           // ADD
       "readingCompleted": true,
     ]
   )
   ```

5. `apps/ios/.../Views/WriteView.swift` — Also fix the `sendUserInput` race by sending the specific dayIndex (or remove it since SyncService handles the text anyway). The simplest fix: don't send `sendUserInput` at all, since the `updateEntry` call already sends the `writingText`, and `addUserInputMemory` for Mem0 can be triggered from the entries route when `writingCompleted` is set.

   Actually, the simpler approach: just move the Mem0 memory call into the `PUT /api/entries` handler when `writingCompleted=true` and `writingText` is present (it already does this for free writes at line 198-206). Then remove the `sendUserInput` call from `completeWriting()`.
