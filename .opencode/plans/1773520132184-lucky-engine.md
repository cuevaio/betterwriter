# Fix: Stream Resume on App Reopen + "Done Reading" Button

## Problem Summary

**Issue 1 — Stream not resuming on reopen:** When the user kills the app during
active SSE streaming and reopens, they see a loading spinner for several seconds,
then the full text appears all at once. The text should appear immediately
(partial draft) and continue streaming.

**Issue 2 — "Done Reading" button missing:** After streaming completes and the
full text is displayed, the "DONE READING" button does not appear.

## Root Causes

### Issue 1
- No partial text is persisted during streaming — everything is in-memory only.
- On reopen, a fresh network request must complete before any text displays.
- The server replays all events from Redis start, but the client shows them via
  rapid typewriter animation that looks like "all at once."

### Issue 2
- After `consumeReadingStream` saves `readingBody` via `saveReadingLocally()`,
  `isLoading` is already `false` (set at first delta). No `@State` change occurs
  after the save to force SwiftUI to re-evaluate the button condition.
- The button depends on `entry?.readingBody != nil` via `@Query`, which updates
  asynchronously. Without a state change to trigger re-render, the condition
  may not be re-evaluated.

## Solution Design

### Approach
1. **Persist partial text** via a new `readingBodyDraft: String?` field on
   `DayEntry`. Save periodically during streaming (debounced, every ~20 deltas).
2. **Instant draft restoration** on reopen: show `readingBodyDraft` immediately
   (no loading spinner), then reconnect to server.
3. **Text-length catch-up** during stream replay: skip typewriter for deltas
   that correspond to already-displayed draft text, then animate only new deltas.
4. **Explicit `streamComplete` flag** (`@State`) to reliably show the button,
   decoupled from `@Query` reactivity timing.

### Catch-Up Algorithm (text-length based)
```
resumeLength = readingBodyDraft.count
streamedBody = ""    // accumulate ALL replayed + new deltas
catchingUp = (resumeLength > 0)

FOR EACH delta(text, _):
  streamedBody += text

  IF streamedBody.count <= resumeLength:
    CONTINUE  // still replaying known text, skip typewriter

  IF catchingUp:
    catchingUp = false
    // Only animate the NEW portion of this delta
    overlapInDelta = max(0, resumeLength - (streamedBody.count - text.count))
    newText = text.dropFirst(overlapInDelta)
    typewriter.enqueue(newText)
  ELSE:
    typewriter.enqueue(text)  // fully live, animate everything

  isLoading = false
```

This approach is stream-ID-agnostic — it works whether the server replays the
same stream or a new one was started.

## Edge Cases

| Case | Handling |
|---|---|
| App killed before any draft save | No draft exists. Behaves as today — full stream. |
| App killed after draft, server completed | POST returns 200 JSON. saveReadingLocally clears draft. |
| App killed after draft, server still running | POST returns 202 SSE. Catch-up skips seen text. |
| Server regenerated (new stream, different text) | Text-length catch-up degrades gracefully. |
| Foreground return with dead URLSession | Detected by foreground handler, triggers reconnect. |
| Draft save modelContext.save() fails | Logged. On next reopen, no draft → full reload. |
| Day advances while draft exists | New dayIndex → different entry → draft ignored. |
