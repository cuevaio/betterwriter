# Fix: "Done Reading" button does nothing + duplicate entries + wrong text

## Problem Summary

After the previous commit (03d9aa3), three interrelated bugs remain:

1. **"Done Reading" button does nothing** — tapping it has no effect, the app
   doesn't transition to the writing screen.
2. **Different text on second open** — closing and reopening the app shows a
   different reading instead of the cached one.
3. **Local-first caching not working** — completed readings aren't reliably
   persisted for instant display on subsequent opens.

## Root Cause Analysis

All three bugs share a single root cause: **ReadView uses a `@Query`-derived
computed property `entry` for all data operations, but `@Query` updates
asynchronously.** This causes a cascade of failures:

### Bug chain

1. When streaming completes, `saveReadingLocally()` calls
   `entry ?? DayEntry(dayIndex:)`. Since `entry` comes from `@Query` and the
   query hasn't refreshed, `entry` is nil → a **new DayEntry is created**.

2. Earlier, `saveDraftImmediately()` (called every 20 deltas) did the same
   thing — `entry` was nil, so it created **another** DayEntry. Now there are
   **duplicate DayEntry objects** with the same dayIndex.

3. `streamComplete` is set to `true` (a `@State` that updates synchronously),
   so the button appears. But `completeReading()` does
   `guard let entry = entry else { return }` — `entry` (from `@Query`) is
   **still nil** → function returns silently → `onComplete()` is never called
   → no phase transition.

4. On second app open, `@Query` returns both duplicate entries. It picks the
   first one, which may be the draft-only entry (no `readingBody`). Since no
   local `readingBody` is found, the app goes to the server.

5. The server sees the normal reading already exists in its DB, so
   `resolveReadingDayIndex()` returns a **bonus reading** dayIndex (100,000+).
   PrefetchStore gets this bonus reading (different text). ReadView consumes it
   and saves it to an entry with the bonus dayIndex — which ReadView's `entry`
   computed property (filtering by the view's dayIndex) will never find.

6. The user sees different text and the button still doesn't work.

### Why WriteView doesn't have this bug

WriteView uses `@State private var managedEntry: DayEntry?` and a
`resolveEntry()` method that does a **synchronous `modelContext.fetch`** (not
`@Query`). This avoids the async timing issue entirely. ReadView should use
the same pattern.

## Solution

Replace ReadView's `@Query`-derived `entry` computed property with a `@State`
direct reference, using a `resolveEntry()` method identical in spirit to
WriteView's. This single change fixes all three bugs:

- **Button works:** `completeReading()` uses the `@State` reference (always
  non-nil after first resolve).
- **No duplicates:** All save operations reuse the same entry object.
- **Correct local caching:** The entry is created with the view's dayIndex and
  persists correctly across app restarts.

## Files to Modify

### 1. `apps/ios/betterwriter/betterwriter/Views/ReadView.swift`

**Add `@State` entry reference:**
```swift
@State private var localEntry: DayEntry?
```

**Add `resolveEntry()` method** (synchronous `modelContext.fetch`, matching
WriteView's pattern):
```swift
@MainActor
private func resolveEntry() -> DayEntry {
  if let existing = localEntry { return existing }

  let targetIndex = dayIndex
  let descriptor = FetchDescriptor<DayEntry>(
    predicate: #Predicate<DayEntry> {
      $0.dayIndex == targetIndex
        && $0.isFreeWrite == false
        && $0.isBonusReading == false
    }
  )
  if let fetched = try? modelContext.fetch(descriptor).first {
    localEntry = fetched
    return fetched
  }

  let entry = DayEntry(dayIndex: dayIndex)
  modelContext.insert(entry)
  do { try modelContext.save() } catch {
    print("ReadView: save new entry failed: \(error)")
  }
  localEntry = entry
  return entry
}
```

**Update `loadReading()`:**
- Step 0: Use `resolveEntry()` to get the entry. Check its `readingBody`.
  If non-nil → show, set `streamComplete`, return.
- Step 1: Check `resolveEntry().readingBodyDraft` for partial text restoration.
- Remove all uses of the computed `entry` property.

**Update `saveDraftImmediately(body:)`:**
```swift
let entry = resolveEntry()
entry.readingBodyDraft = body
do { try modelContext.save() } catch { ... }
```

**Update `saveReadingLocally(body:, dayIndex:)`:**
- Remove the `dayIndex` parameter (always use the view's dayIndex via
  `resolveEntry()`).
```swift
let entry = resolveEntry()
entry.readingBody = body
entry.readingBodyDraft = nil
do { try modelContext.save() } catch { ... }
```

**Update `completeReading()`:**
```swift

entry.needsSync = true
...
onComplete()
```

**Update foreground handler:**
- Use `localEntry` or `resolveEntry()` instead of the computed `entry`.

**Update `.task` initializer:**
- Use `resolveEntry()` for the initial streamComplete check.

**Keep computed `entry` only for the view body fallback** (line 36):
```swift

```
This is a display-only fallback and is fine to keep using `@Query`.
