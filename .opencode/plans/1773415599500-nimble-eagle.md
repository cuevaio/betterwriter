# Fix: Bonus readings always the same + blank screen after completion

## Root Cause

The entity lock (`stream:active:{userId}:reading`) persists for 24h after generation completes. Both daily and bonus readings share the same lock key. The POST handler's "completed" branch returns whichever entry the lock points to, without checking if the user still needs it.

### Original bug: same bonus reading every time

Lock from bonus 100000 stays after completion → next POST returns the same bonus entry.

### Regression from v2 fix (hoisting `resolveReadingDayIndex`): blank screen

Hoisting `resolveReadingDayIndex` BEFORE the lock check is wrong because that function looks FORWARD: if the current day already has a `readingBody`, it returns a bonus dayIndex. This breaks the reconnection scenario:

1. Daily reading generation starts for day 1. User closes app mid-stream.
2. Server completes generation → `readingBody` saved for day 1 → lock stays.
3. User re-opens → `resolveReadingDayIndex` sees day 1 has reading → returns bonus 100000.
4. Lock check: `lockedDayIndex (1) ≠ 100000` → releases lock → starts generating BONUS.
5. ReadView (expecting day 1) streams the bonus content. On `complete`, `displayedText` is cleared, server entry has dayIndex 100000, ReadView's `entry` filter (`dayIndex == 1`) finds nothing → **blank screen**.

This is confirmed by the server logs: POST returns 202 (new generation started) when it should have returned 200 (completed, return existing entry).

## Correct Fix

### Fix 1 — Use `readingCompleted` check WITHOUT hoisting (revert v2, restore v1)

**File:** `apps/web/app/api/readings/generate/stream/route.ts`

Move `resolveReadingDayIndex` back to AFTER the lock check (its original position). In the "completed" branch, use the `readingCompleted` DB field:

- `readingCompleted = false` → user hasn't finished reading → return the entry (handles reconnection)
- `readingCompleted = true` → user is done with this reading → release lock → fall through to generate new reading

```ts
// Step 1: Check for an existing active stream
const active = await resolveActiveStream(userId, "reading");
if (active) {
  if (active.meta.status === "running") {
    return Response.json({ ok: true, mode: "already-running" });
  }
  if (active.meta.status === "completed") {
    const lockedDayIndex = active.meta.params.dayIndex as number;
    const rows = await db.select()...where(dayIndex = lockedDayIndex)...limit(1);
    if (rows.length > 0 && rows[0].readingBody !== null) {
      if (rows[0].readingCompleted) {
        // User finished this reading — release lock, generate next
        await releaseEntityLock(userId, "reading", active.streamId);
      } else {
        // User hasn't finished — return the entry (reconnection case)
        return Response.json({ mode: "completed", entry: {...} });
      }
    } else {
      await releaseEntityLock(userId, "reading", active.streamId);
    }
  }
}

// Step 2: resolveReadingDayIndex called HERE (after lock check)
const { dayIndex, isBonusReading, currentDayIndex } =
  await resolveReadingDayIndex(userId);
```

**Why this works for all scenarios:**

| Scenario | readingCompleted | Action | Result |
|---|---|---|---|
| Close mid-stream, re-open | false | Return entry | Correct: client gets its daily reading |
| Bonus completed, want new bonus | true (synced) | Release lock | Correct: generates new bonus |
| Bonus completed, need daily reading | true (synced) | Release lock | Correct: generates daily |
| Sync failed, any case | false | Return old entry | Acceptable: SyncService retries on next foreground |
